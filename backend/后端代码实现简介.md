## 架构设计

```
HTTP 请求 → Drogon 路由 → Controller → DbClient (MySQL) → JSON 响应
                                ↓
                          models/ 数据模型（序列化）
```

- **路由注册**：通过 Drogon 的 `METHOD_LIST_BEGIN` / `ADD_METHOD_TO` 宏声明式注册，支持路径参数（`{genealogy_id}`、`{member_id}`）
- **数据库访问**：`app().getDbClient()->execSqlSync()` 同步执行 SQL，Drogon 框架保证非阻塞 I/O
- **认证鉴权**：Bearer Token 机制，`AuthController` 管理内存 Token 存储，各 Controller 通过 `getUserIdFromToken()` 验证身份
- **权限控制**：族谱创建者（owner）+ 共享权限（editor/viewer）三级权限模型

## 核心性能优化

### 1. 祖先查询：Materialized Path

**原理**：每行存储 `ancestor_path` 字段（如 `/1/862/2717/33332`），查询祖先时直接解析该字段，无需递归 CTE。

```
优化前: 递归 CTE（深度30代超时）
优化后: SELECT ancestor_path FROM Member WHERE member_id=? → 解析字符串 → 批量查询
性能: 从超时 → 毫秒级
```

### 2. 后代查询：双层策略

- **根节点快速通道**：利用 `idx_member_genealogy_gen(genealogy_id, generation)` 复合索引，一条 SQL 范围扫描

```sql
SELECT ... FROM Member
WHERE genealogy_id = ? AND generation > ? AND generation <= ?
ORDER BY generation
```

- **通用迭代通道**：利用 `father_id`/`mother_id` 索引逐层 IN 查询，每层精确命中

```
优化前: 递归 CTE（深度30代超时2分钟+）
优化后: 根节点 48K 后代仅需 1.88s（SQL ~50ms，其余为 JSON 序列化）
```

### 3. 亲缘关系查询：批量替代 N+1

**原理**：收集所有需要的成员 ID → 一次 `WHERE member_id IN (...)` 批量查询 → 内存 map 查找。

```
优化前: 1065 次 SQL（单独查询每个成员 + 配偶）
优化后: 3~4 次 SQL
```

### 4. 其他优化

- **全文搜索**：MySQL FULLTEXT 索引替代 `LIKE '%keyword%'`，支持中文分词
- **统计缓存**：内存 TTL 缓存（`StatsCache`），避免重复计算
- **批量接口**：`/api/member/batch` 支持一次查询最多 5000 个成员
- **精确列选择**：避免 `SELECT *`，不传输 TEXT 类型的 `biography` 字段

## 数据库设计

### 核心表

| 表名 | 说明 | 关键字段 |
|------|------|----------|
| `User` | 用户表 | user_id, username, password |
| `Genealogy` | 族谱表 | genealogy_id, name, surname, user_id |
| `Member` | 成员表 | member_id, father_id, mother_id, spouse_id, generation, ancestor_path |
| `Marriage` | 婚姻表 | marriage_id, husband_id, wife_id |
| `GenealogyShare` | 共享表 | share_id, genealogy_id, user_id, permission |
| `StatisticsCache` | 缓存表 | cache_key, cache_value (JSON), expires_at |

### 关键索引

- `idx_member_genealogy_gen(genealogy_id, generation)` — 根节点后代快速通道
- `idx_member_ancestor_path(ancestor_path)` — 路径前缀查询
- `ft_member_name(name)` — 全文搜索
- `idx_member_father_id(father_id)` / `idx_member_mother_id(mother_id)` — 父子关系查询

## 构建运行

### 环境要求

- **系统**：Linux（推荐 Ubuntu 20.04+）/ WSL2
- **编译器**：GCC 9+ 或 Clang 10+（需支持 C++17）
- **依赖**：
  - [Drogon](https://github.com/drogonframework/drogon) Web 框架
  - jsoncpp
  - MySQL 8.0+
  - CMake 3.5+

### 安装依赖

```bash
# Ubuntu/Debian
sudo apt install build-essential cmake libjsoncpp-dev

# 安装 Drogon（从源码）
git clone https://github.com/drogonframework/drogon.git
cd drogon && mkdir build && cd build
cmake .. && make -j$(nproc) && sudo make install

# 安装 MySQL 客户端库
sudo apt install libmysqlclient-dev
```

### 构建

```bash
cd backend
mkdir build && cd build
cmake ..
cmake --build .
```

### 配置

创建 `config.json`（参考结构）：

```json
{
    "listeners": [
        {
            "address": "0.0.0.0",
            "port": 8088
        }
    ],
    "db_clients": [
        {
            "name": "default",
            "rdbms": "mysql",
            "host": "127.0.0.1",
            "port": 3306,
            "dbname": "genealogy_db",
            "user": "root",
            "password": "",
            "client_encoding": "utf8mb4"
        }
    ],
    "app": {
        "number_of_threads": 4
    }
}
```

### 运行

```bash
cd backend/build
./genealogy_system
# 服务启动在 http://localhost:8088
```

### 数据库初始化

```bash
# 1. 创建数据库
mysql -u root -e "CREATE DATABASE genealogy_db CHARACTER SET utf8mb4;"

# 2. 执行建表迁移
mysql -u root genealogy_db < migrations/migration_phase3.sql

# 3. 修复辈分数据（如从 CSV 导入后）
mysql -u root genealogy_db < migrations/migration_fix_generations.sql
```

## 安全设计

- **认证**：Bearer Token 内存管理，支持登出失效
- **密码**：SHA-256 哈希存储（通过 `encrypt_password`/`decrypt_password`）
- **权限**：族谱级三级权限（owner > editor > viewer），成员操作继承族谱权限
- **SQL 注入防护**：全部使用参数化查询（`?` 占位符），杜绝拼接 SQL
- **约束**：数据库层 CHECK 约束（辈分 > 0、出生日期 ≤ 死亡日期、ancestor_path 格式）
