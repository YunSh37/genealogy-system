## 1. 整体概览

### 这是什么？

一个在浏览器中运行的**族谱管理工具**。用户无需安装任何软件，打开网页即可：

- 创建和管理多个族谱
- 添加、编辑、搜索家族成员
- 以**树形图**直观查看家族后代关系
- 查询某人的祖先链或后代
- 查看家族统计数据（男女比例、辈分分布等）

### 用什么做的？

| 层面 | 技术 | 说明 |
|------|------|------|
| 页面结构 | HTML | 只用一个页面（`index.html`），所有内容动态切换 |
| 外观样式 | CSS | 中国传统配色，暖色调为主 |
| 交互逻辑 | JavaScript（原生） | 没有用 Vue/React 等框架，纯手写 |
| 树形图 | Canvas 2D | HTML5 画布，像素级绘制每个节点和连线 |

> **为什么不用框架？** 这个项目功能明确、模块独立，纯 JS 完全可以胜任，还能避免框架带来的打包、更新等额外复杂度。对于不熟悉前端框架的用户，纯 JS 代码也更容易理解和修改。

## 3. SPA 单页应用架构

### 什么是 SPA？

**SPA（Single Page Application，单页应用）** 的意思是：整个网站只有一个 HTML 页面，切换不同"页面"时，不是跳转到新网址，而是用 JavaScript 把当前区域**隐藏/显示**。

### 本项目的页面切换

```
index.html 中包含所有"页面"的 HTML：

┌─ #view-login ──────────────┐  ← 登录/注册页
├─ #view-main ───────────────┤  ← 主界面（登录后显示）
│   ├─ #page-genealogy-list  │  ← 族谱列表
│   └─ #page-genealogy-workspace  ← 族谱工作区
│       ├─ #sub-dashboard    │  ← 仪表盘
│       ├─ #sub-family-tree  │  ← 族谱树
│       ├─ #sub-members      │  ← 成员管理
│       ├─ #sub-ancestors    │  ← 祖先查询
│       ├─ #sub-descendants  │  ← 后代查询
│       ├─ #sub-relationship │  ← 亲缘关系
│       ├─ #sub-statistics   │  ← 统计分析
│       └─ #sub-share        │  ← 族谱共享
└────────────────────────────┘
```

每次"切换页面"，实际上是：
1. 把当前显示的页面加上 `display: none`（隐藏）
2. 把目标页面加上 `display: block`（显示）

关键代码在 `app.js` 中：

```javascript
// 切换顶层页面（登录 ↔ 主界面）
_showPage(name) {
  document.querySelectorAll('.view').forEach(p => p.classList.remove('active'));
  document.getElementById('view-' + name).classList.add('active');
}

// 切换子页面（仪表盘 ↔ 族谱树 ↔ 成员管理 ...）
_showSubPage(name) {
  document.querySelectorAll('.sub-page').forEach(p => p.classList.remove('active'));
  document.getElementById('sub-' + name).classList.add('active');
  this._loadSubPageData(name);  // 切过去时加载数据
}
```

## 4. API 客户端（api.js）

### 职责

封装所有与后端服务器的 HTTP 请求，让 `app.js` 可以像调用本地函数一样获取数据。

### 设计

```javascript
const API = {
  // 基础请求方法（私有）
  _get(url, params)     // GET  请求
  _post(url, data)      // POST 请求
  _put(url, data)       // PUT  请求
  _delete(url)          // DELETE 请求

  // 业务接口（公开）
  login(username, password)            // 登录
  register(username, password)         // 注册
  getGenealogies()                     // 获取族谱列表
  createGenealogy(name)               // 创建族谱
  getMembers(gid, page, size)         // 分页获取成员列表
  getAllMembers(gid)                   // 获取全部成员
  getDescendants(mid, depth)          // 查询后代
  getFamilyTree(mid, depth)           // 查询族谱树（祖先+自己+后代）
  getAncestors(mid, depth)            // 查询祖先
  getRelationship(id1, id2)           // 查询亲缘关系
  // ... 更多接口
}
```

### 关键细节

- **Token 管理**：登录成功后 Token 存 `localStorage`，每次请求自动附带在 `Authorization` 头
- **401 自动跳转**：Token 过期或无效时，自动跳回登录页
- **错误统一处理**：网络错误和业务错误都转为统一的错误消息

### 5.4 族谱树缓存机制

**问题**：用户从族谱树切换到成员管理再切回来，每次都要重新调用 API 加载数据，体验差。

**解决**：两级缓存机制。

```
用户切回"族谱树"页
        │
        ▼
  缓存中有数据吗？ ──否──→ 正常加载（调 API + 渲染）
        │
       是
        │
        ▼
  族谱 ID + 参数一致吗？ ──否──→ 正常加载
        │
       是
        │
        ▼
  画布还在页面上吗？ ──是──→ 直接显示（瞬间完成 ⚡）
        │
       否（被其他模块销毁了）
        │
        ▼
  用缓存数据重建画布（跳过 API ⚡）
```

**缓存失效条件**（以下任一发生就清除缓存）：
- 切换族谱
- 用户退出登录
- 用户点击"加载"按钮（主动强制刷新）

### 5.5 智能根节点查找

族谱树默认显示时，需要自动选一个合适的"根"（树的最顶层节点）。

**旧方案**：找"第 1 代男性" → 有问题，族谱中可能存在无关联的第 1 代孤立成员。

**新方案**：优先级排序算法

```
获取族谱全部成员
    ↓
找出"有子女"的成员（排除孤立节点）
    ↓
按三层优先级排序，取最优：
  ① 无父母者优先（真正的始祖）
  ② 辈分小的优先
  ③ ID 小的优先
    ↓
兜底：如果所有人都是孤立的，取 ID 最小的成员
```

正常模式                      全屏模式
┌──────────────────┐         ┌──────────────────────┐
│ 工具栏           │         │ 全屏工具栏（顶部固定） │

## 6. 族谱树渲染引擎（tree-renderer.js）

这是整个项目**技术含量最高**的模块。它不依赖任何第三方库，只用 Canvas 2D API 在浏览器画布上像素级绘制族谱树。

### 6.1 为什么用 Canvas 而不是 DOM？

| 方案 | 100 个节点 | 1000 个节点 | 10000 个节点 |
|------|-----------|------------|-------------|
| DOM（div 排列） | ✅ 流畅 | ⚠️ 开始卡 | ❌ 浏览器崩溃 |
| Canvas 2D | ✅ 流畅 | ✅ 流畅 | ⚠️ 需要优化 |

族谱树可能包含数千个节点，Canvas 是唯一可行的方案。

### 6.2 树布局算法

这是最核心的算法，决定了每个节点放在画布的什么位置。

#### 节点尺寸

```
┌──────────┐
│ 第 X 代  │ ← 代际标签（顶部）
│          │
│   张     │
│          │  ← 姓名（竖排，逐字绘制）
│   三     │
│          │
│  1890    │ ← 出生年份（底部）
│  ▼ 折叠  │ ← 折叠标记（可折叠时显示）
└──────────┘
  宽 56px
  高 82px
```

#### 布局步骤

```
步骤一：建树（buildTree）
─────────────────────────
把扁平的成员列表 → 组织成树状结构

输入：[{id:1, father_id:null}, {id:2, father_id:1}, {id:3, father_id:1}, ...]
输出：
  根节点1
  ├── 节点2
  │   ├── 节点4
  │   └── 节点5
  └── 节点3
      └── 节点6


步骤二：计算子树宽度（后序遍历）
────────────────────────────────
从叶子节点向上计算每个子树需要多宽

叶子节点：宽 = NODE_W + H_GAP (= 68px)
有多个孩子的节点：宽 = 所有子树宽度之和
有配偶的节点：额外留出配偶节点空间


步骤三：分配坐标（先序遍历）
────────────────────────────
从根节点开始向下分配 (x, y) 坐标

根节点：y = 20（顶部留白）
子节点：y = 父节点.y + NODE_H + V_GAP (= 父节点.y + 192)
配偶节点：与配偶并排，水平偏移
```

#### 关键间距参数

```
H_GAP = 12    水平间距（同代兄弟节点之间）
V_GAP = 110   垂直间距（父子代之间，必须 > NODE_H=82 防止重叠）
```

### 6.3 折叠/展开机制

每个节点可以折叠或展开其子节点。

```
展开状态（▼）              折叠状态（▶）
   节点A                     节点A
    ├── B                     ▶ (3个后代已折叠)
    │   ├── D
    │   └── E
    └── C
```

**实现要点**：
- 每个节点有一个 `_collapsed` 属性
- 折叠时：递归隐藏所有子节点
- 展开时：**只展开一代**（不全部展开，避免一次性显示几万个节点）
- 双击节点触发折叠/展开
- 重新布局时，被折叠的子树不出现在渲染数组中

**自动折叠**：树深度 ≥ 8 代时，自动折叠第 7 代及以下的节点，让初始视图保持清爽。

**一键展开**：📖 按钮递归展开所有节点（适用于节点数较少的情况）。

### 6.4 渲染优化

#### 视口裁剪（节点级别）

只绘制当前可视区域内的节点：

```
整个树（可能几万个节点）
┌──────────────────────────────────────┐
│                                      │
│    ┌──────────┐                      │
│    │ 屏幕可见  │  ← 只渲染这个区域    │
│    │ 区 域    │    的节点            │
│    └──────────┘                      │
│                                      │
└──────────────────────────────────────┘
```

连线不裁剪，确保从屏幕外的父节点到屏幕内子节点的连线仍然可见。

#### 渲染合并（rAF 批处理）

滚轮缩放、拖拽平移等高频事件不会每次都触发绘制，而是用 `requestAnimationFrame` 合并为一次渲染：

```
滚轮事件1 ─┐
滚轮事件2 ─┤→ 标记 _renderScheduled = true
滚轮事件3 ─┘
              │
              ▼
     下一帧到来时（约 16ms 后）
              │
              ▼
       执行一次 _doRender()
```

| 操作 | 实现 |
|------|------|
| 滚轮缩放 | 以鼠标位置为中心缩放（0.02× ~ 3.0×） |
| 拖拽平移 | mousedown → mousemove → mouseup 跟踪 |

### 6.6 连接线绘制

三种连接关系：

```
配偶连接（紫色，水平虚线）
  夫 ═══════ 妻

父子/母子连接（棕色，曲线）
  父               母（如果没有父亲）
   │               │
   ├────┬────┐     ├────┬────┐
   子1  子2  子3   子1  子2  子3
```

- **父优先原则**：孩子优先连接到父亲，父亲不在族谱中时才连到母亲
- **配偶双向**：夫妻之间只需画一次连线（由 ID 较小的一方负责绘制）
- **连线不裁剪**：即使父节点在屏幕外，到子节点的连线仍然画出

## 8. 关键技术决策

### 为什么不用框架？

| 考虑因素 | 决策 |
|---------|------|
| 项目规模 | 约 4000 行 JS，框架的收益小于学习成本 |
| 性能需求 | Canvas 渲染是瓶颈，框架帮不上忙 |
| 可维护性 | 纯 JS 代码逻辑直接，新人容易上手 |
| 部署复杂度 | 零构建步骤，改完即生效 |

### 为什么 TreeRenderer 是单例？

`TreeRenderer` 设计为全局唯一的渲染器实例（不是 Class），原因：

1. **Canvas 上下文只能同时存在一个活跃实例** — 浏览器中同一时刻只有一个族谱树在显示
2. **避免状态混乱** — 单例确保节点数据、缩放状态、事件绑定不会混淆
3. **简化调用** — 直接 `TreeRenderer.init()` 而不是 `new TreeRenderer()`

代价是模块间会互相影响（如后代查询会销毁族谱树渲染器），解决方案是**数据缓存 + 按需重建**。

### 为什么默认用后代接口而不是族谱树接口？

| 接口 | 返回数据 | 适用场景 |
|------|---------|---------|
| `getDescendants` | 纯后代树 | 默认显示（清晰的自顶向下关系） |
| `getFamilyTree` | 祖先 + 自己 + 后代 | 用户指定成员 ID 时（完整上下文） |

默认使用后代接口可以避免祖先路径混入导致的拓扑异常。

## 9. 数据流全景

### 用户操作 → 屏幕更新的完整路径

```
用户在页面点击/输入
        │
        ▼
app.js 中的事件处理函数
        │
        ▼
调用 API.xxx() 发送 HTTP 请求
        │
        ▼
Flask 代理转发到 Drogon 后端
        │
        ▼
后端查询数据库，返回 JSON
        │
        ▼
api.js 解析 JSON，返回给 app.js
        │
        ▼
app.js 处理数据
        │
        ├── 列表/统计类：直接更新 DOM
        │
        └── 树形图类：交给 tree-renderer.js
              │
              ▼
            buildTree()  平铺数据 → 树结构
              │
              ▼
            _layout()    树结构 → (x, y) 坐标
              │
              ▼
            _doRender()  坐标 → Canvas 像素
              │
              ▼
            (屏幕上出现树形图)
```

### 模块间依赖关系

```
api.js        ← 无依赖，纯工具
tree-renderer.js ← 无依赖（仅依赖 Canvas API）
app.js        ← 依赖 api.js + tree-renderer.js
index.html    ← 依赖以上全部（通过 <script> 标签加载）
style.css     ← 无依赖
```
